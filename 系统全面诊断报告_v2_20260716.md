### 1.3 核心数据流

```
开单流程:
  输入表单 → 创建工单 → 同步设备明细 → 扣减库存 → 生成工资记录 → 返回工单

收款流程:
  录入收款 → 更新工单paid_amount → 更新工单payment_status → 记录日志

工资结算:
  工单工资记录 → 标记已结算 → 财务统计引用

删除流程:
  删除工单 → 清理照片 → 清理设备明细 → 清理工资记录 → 重置收款关联 → 清理待办关联
```

### 1.4 权限模型

| admin | 全量数据，所有操作，所有统计 |
| worker | 仅可查看/操作自己相关的工单、工资，统计数据按权限过滤 |

**功能**: 首页仪表盘，展示今日数据、本月统计、待办、预警等

**API**: `GET /api/dashboard`
**权限**: 已修复（v2），财务数据已按角色过滤

- P1: unpaid_salary 对 worker 仅显示自己的

### 2.2 工单管理 (施工/维修)

**功能**: 创建、编辑、删除、查询施工记录和维修工单

**API**: 6个端点
- `GET /api/records` - 查询列表
- `POST /api/records` - 创建工单
- `GET /api/records/<id>` - 获取详情
- `PUT /api/records/<id>` - 更新工单
- `DELETE /api/records/<id>` - 删除工单
- `GET /api/records/<id>/edits` - 编辑历史

**当前状态**: ✅ 已修复核心Bug
- Bug1（材料费丢失）已修复：材料和设备费用分离统计
- Bug2（工资全量删重建）已修复：仅删除未结算记录
- Bug3（删除不清理关联）已修复：级联清理关联数据
- Bug6（库存不足无告警）已修复：API返回warnings，前端弹窗提示

**仍存在的问题**:
1. **施工状态与维修状态混用同一个status字段**: 施工有6个状态(pending→in_progress→settlement→completed)，维修有9个状态(pending→dispatched→in_progress→callback→settlement→completed)，两者共用同一个字段，前端可能显示错误的状态标签
2. **批量删除未清理关联数据**: `batch_operation_records` 的 delete 操作只删除了照片，没有清理 RepairEquipment、SalaryRecord、PaymentRecord 等关联数据
3. **工单编辑弹窗中设备明细不显示**: 编辑工单时，设备明细需要单独加载显示
4. **fee_items 与 equipment_details 双重数据结构**: 费用明细既存在 fee_items JSON 中，又通过 equipment_details 同步到 RepairEquipment 表，数据冗余且容易不一致

**表单字段分析** (workForm):

| 分类 | 字段 | 必要性 | 诊断 |
|------|------|--------|------|
| 基础信息 | 记录类型(施工/维修) | ✅ 必须 | |
| 客户信息 | 客户名称、联系人、电话 | ✅ 必须 | |
| 工作信息 | 工作地址、工作日期、状态、优先级 | ✅ 必须 | |
| 类型细分 | 施工类型/工单类型(work_subtype) | ✅ 必须 | |
| 时间 | 开始时间、结束时间、工时 | ⚠️ 可选 | 维修场景有意义，部分施工不需要 |
| 人员 | 固定员工多选、临时工明细 | ✅ 必须 | 目前OK |
| 费用 | 人工费、材料费、交通费、其他费 | ✅ 必须 | |
| 费用明细 | fee_items 表格 | ⚠️ 冗余 | 与上方费用字段重复，保留一个即可 |
| 设备明细 | equipment_details 表格 | ✅ 必须 | 维修必须，施工可选 |
| 维修专用 | 故障描述、诊断、维修过程、结果 | ⚠️ 条件必须 | 维修时必填，施工时隐藏 |
| 维修专用 | 未完成原因类型、原因说明 | ⚠️ 条件必须 | 维修结果=未完成时必填 |
| 税费 | 税种(含税/不含税)、税率 | ✅ 必须 | |
| 付款 | 付款状态、已付金额 | ⚠️ 冗余 | 付款状态应由收款记录自动计算，不应手动修改 |
| 保修 | 保修天数 | ⚠️ 可选 | |
| 照片 | 工作照片上传 | ✅ 必须 | |
| 备注 | 备注 | ⚠️ 可选 | |
| 满意度 | 客户反馈、满意度 | ⚠️ 可选 | 实际很少填写 |

**冗余字段**: fee_items明细表与上方固定费用字段重复；payment_status和paid_amount不应在工单表单中手动填写

**缺失字段**: 工单关联项目(project_id)在表单中不可见，只能通过项目入口创建

### 2.9 项目管理 (Projects)

**功能**: 项目/合同管理，包含项目工单、项目支出、项目工资

**数据模型**: 
- Project（项目主表）
- ProjectWorkRecord（项目施工记录，独立于WorkRecord）
- ProjectExpense（项目支出）
- ProjectSalary（项目工资）

**API**: 约15个端点（项目CRUD + 子记录CRUD + 阶段管理）

**当前状态**: ⚠️ 架构存在设计问题

**严重问题**:
1. **项目工单与普通工单是两套独立表**: ProjectWorkRecord 和 WorkRecord 是完全独立的，这意味着：
   - 项目工单不参与仪表盘统计
   - 项目工单不出现在工单查询列表
   - 项目工单没有设备明细、工资同步等逻辑
   - 两套代码维护成本翻倍
2. **WorkRecord 有 project_id 字段但未充分利用**: 普通工单可以通过 project_id 关联项目，但前端没有入口
3. **项目与普通工单的收款不统一**: 项目收款通过 receipt_amount 字段记录，不走 PaymentRecord 表
4. **项目删除时未清理关联的普通工单**: WorkRecord 的 project_id 设为 NULL，但工单内容未处理

**表单字段分析** (projectForm):

| 字段 | 必要性 | 诊断 |
|------|--------|------|
| 项目名称 | ✅ 必须 | |
| 项目编号 | ✅ 自动生成 | |

## 四、架构优化建议

### 4.1 短期优化（1-2周可实现）

1. **统一项目工单与普通工单**: 将 ProjectWorkRecord 合并到 WorkRecord，通过 project_id 区分，避免双表维护
2. **完善权限过滤**: 对所有列表/统计API统一应用权限过滤
3. **简化工单表单**: 移除手动 payment_status/paid_amount 字段，移除 fee_items 与固定费用字段的重复

### 4.2 中期优化（2-4周）

1. **通知系统激活**: 在库存预警、维护到期、收款逾期等场景自动创建通知
2. **巡检计划自动化**: 添加定时任务，自动按周期生成待办
3. **工资管理完善**: 支持固定员工工资记录、部分结算、独立编辑删除
4. **物料-支出联动**: 采购入库时自动生成支出记录

### 4.3 长期优化

1. **前后端分离重构**: 将 routes.py 按模块拆分（当前4150行单文件）
2. **移动端原生体验**: PWA 优化，离线缓存，推送通知
3. **数据仓库**: 引入汇总表，加速统计查询
4. **多租户支持**: 当前架构只支持单团队使用
