### 2.6 🔴 关键架构问题：数据模型重复

**核心问题：普通工单与项目工单分离，导致财务数据分裂**

```
普通业务域                    项目业务域
─────────────────────────────────────────────
WorkRecord         ←→  ProjectWorkRecord   (施工记录)
SalaryRecord       ←→  ProjectSalary       (工资)
Expense            ←→  ProjectExpense      (支出)
(无对应)           ←→  Project (项目本身)
PaymentRecord      ←→  (项目收款独立统计?)
```

**后果**：
1. **统计口径分裂**：经营统计/利润统计只统计WorkRecord+SalaryRecord+Expense，不包含项目域数据
2. **两套工资逻辑**：SalaryRecord和ProjectSalary字段不一致（前者work_units按天，后者支持hourly/daily/piece多种）
3. **两套支出逻辑**：Expense有分类管理（ExpenseCategory），ProjectExpense没有
4. **API重复**：项目域下有自己的records/expenses/salaries CRUD接口，与全局接口平行
5. **收款不统一**：PaymentRecord是全局的，但项目的receipt_amount/warranty_amount存在Project表中

### 3.4 工单管理模块（核心，WorkRecord）

#### 3.4.1 功能逻辑

工单是系统核心，支持两种业务类型：

**施工记录(construction)**：
- 流程：创建 → 施工中 → 结算 → 完成
- 状态流：pending → in_progress → settlement → completed
- 关键字段：work_content（施工内容）、费用明细、参与员工

**维修工单(repair)**：
- 流程：报修 → 派单 → 维修中 → 回访 → 结算 → 完成
- 状态流：pending → dispatched → in_progress → callback → settlement → completed
- 特殊状态：unable（无法维修）、cancelled（取消）、rework（返工）
- 关键字段：fault_description（故障描述）、fault_diagnosis（诊断）、repair_process（维修过程）、repair_result（维修结果）

#### 3.4.3 工单模块问题总结

🔴 **严重问题**：
1. **费用字段双轨制**：同时存在labor_fee/material_fee/transport_fee/other_fee四个扁平字段 和 fee_items结构化明细，两套数据可能不一致
2. **repair_equipment.material_id不自动扣库存**：添加工单维修设备时选择了物料，但没有自动创建MaterialStockLog出库记录，库存不会减少
3. **staff_name冗余字段**：staff_names（逗号分隔多选）已存在，staff_name（单人）是兼容旧数据的冗余字段，可能导致权限过滤漏判
4. **payment_status与PaymentRecord数据不一致风险**：工单的paid_amount字段与PaymentRecord表记录的收款总额可能不同步

🟡 **中等问题**：
5. **施工/维修共用表单但字段混用**：施工单不需要故障诊断字段，维修单不需要施工内容以外的冗余字段，当前全显示会造成填写困惑
6. **工单号(order_no)生成规则未统一**：查看代码发现生成逻辑可能不一致
7. **临时工明细(temp_staff_details)JSON**：与staff_names字段有重叠，数据可能不一致
8. **work_photos逗号分隔存储**：无法存储带逗号的文件名；无缩略图管理逻辑在后端（前端生成了但后端没对应清理）

🟢 **建议优化**：
9. **工单状态流转无强制校验**：可以任意跳状态，没有状态机约束（如pending不能直接到completed）
10. **is_completed与status语义重叠**：is_completed是布尔值，status里也有completed状态，两者需保持一致但无数据库约束

### 3.6 巡检计划模块（MaintenancePlan + CustomerEquipment）

- 🔴 **自动巡检未激活**：`generate-todos`是手动接口，没有定时任务/cron调用，必须人工点击
- 🟡 **设备照片字段缺失**：CustomerEquipment模型中删除逻辑引用了equip.photos，但模型中没有photos字段（代码bug）
- 🟡 **完成巡检后未回写设备**：巡检完成后没有更新设备的last_maintenance/next_maintenance字段
- ✅ 删除计划时已处理关联待办（解除related_record_id）

### 3.7 收款管理模块（PaymentRecord）

**功能逻辑**：
- 记录每笔收款：客户、金额、日期、支付方式、是否开票、关联工单(可选)
- 收款统计：按日期/客户/支付方式汇总
- 增删改查+单个工单收款列表

**问题**：
- 🔴 **收款与工单paid_amount不同步**：
  - 添加/修改/删除PaymentRecord时，没有同步更新关联WorkRecord.paid_amount
  - WorkRecord.payment_status也不会自动更新（unpaid→partial→paid）
- 🔴 **删除收款不回滚工单**：删除PaymentRecord时没有回写WorkRecord.paid_amount
- 🟡 **项目收款未独立**：项目的收款(receipt_amount/warranty_amount)存在Project表中，与PaymentRecord无关，统计时无法统一
- ✅ 列表权限已修复：worker只能看到自己相关的收款记录

### 3.8 支出管理模块（Expense + ProjectExpense）

**功能逻辑**：
- 全局支出(Expense)：日常支出、物料采购入库自动生成
- 支出分类(ExpenseCategory)：可管理分类
- 项目支出(ProjectExpense)：项目施工过程中的支出，有收据照片

**问题**：
- 🔴 **两套支出模型分裂**：
  - Expense有expense_type(purchase/salary/daily/other)，ProjectExpense没有
  - Expense关联ExpenseCategory，ProjectExpense只有category文本字段
  - 两套支出在统计/导出时需要分别查询再合并
- 🟡 **删除支出无关联检查**：删除Expense未检查是否由物料入库自动生成（related_type='stock_log'），可能导致库存流水和支出不匹配
- ✅ 权限已修复：worker看不到支出明细；CRUD需admin

### 3.9 工资管理模块（SalaryRecord + ProjectSalary）

**功能逻辑**：
- 普通工资(SalaryRecord)：按工单/业务记录的日薪制工资，支持预支/补贴/扣款
- 项目工资(ProjectSalary)：更灵活，支持时薪(hourly)/日薪(daily)/计件(piece)/预支(advance)/奖金(bonus)
- 结算(settle)：标记已结算，记录结算日期和支付方式

**问题**：
- 🔴 **两套工资模型分裂**：与支出同样的问题，统计口径不统一
- 🔴 **工资与工单的关联弱**：SalaryRecord有business_record_id，但删除工单时没有级联处理工资记录（变成孤立数据）
- 🟡 **固定工月薪没有自动计算**：monthly_salary字段存在但系统没有自动月结功能，需要管理员手动创建工资记录
- 🟡 **ProjectSalary关联work_record_id**：关联的是ProjectWorkRecord的ID，但没有外键约束，容易产生无效ID

### 3.10 物料库存模块（Material + MaterialStockLog）

**功能逻辑**：
- 物料主数据：编号、名称、分类、品牌、型号、规格、单位、单价、库存、预警阈值
- 库存流水：入库(in)/出库(out)/盘点调整(adjust)，记录前后库存、操作人
- 入库可自动生成支出记录（auto_create_expense）
- 低库存预警：stock <= min_stock标记is_low_stock

**问题**：
- 🔴 **RepairEquipment选择物料后不出库**：工单里"维修设备/用料"可选择material_id，但保存工单时不会自动创建出库流水，库存数量不准确
- 🔴 **出库无接口**：只有入库(in)/调整(adjust)，没有专门的出库单/领料单流程
- 🟡 **物料单价历史**：入库时会更新material.unit_price为最新采购价，无法追踪历史采购价格（虽然stock_log里有unit_price，但主表只有当前价）
- 🟡 **无物料分类管理**：category是文本字段，没有独立的分类表，无法对分类排序/启用禁用
- ✅ 权限已修复：worker看不到采购单价(unit_price)和供应商(supplier)

### 3.11 项目管理模块（Project + 子表）

**功能逻辑**：
- 项目基础信息：项目编号、名称、客户、合同信息、金额、工期、负责人、状态
- 项目阶段(project_stage)：准备→备料→施工→完工→验收→结算
- 施工阶段(construction_phase)：配管→穿线→设备安装→设备调试→系统测试
- 计费方式(billing_type)：总价包干(lump_sum)/清包工(labor_only)/计时(hourly)
- 子模块：项目施工记录(ProjectWorkRecord)、项目支出(ProjectExpense)、项目工资(ProjectSalary)

**问题**：
- 🔴 **项目与普通工单关系模糊**：
  - WorkRecord有project_id字段可以关联项目（说明设计上考虑过统一）
  - 但Project又有自己独立的ProjectWorkRecord
  - 两套施工记录并存，用户不知道该用哪个
- 🔴 **项目汇总字段不自动计算**：actual_amount、receipt_amount、warranty_amount需要手动维护，不会从子表自动汇总
- 🟡 **项目成员(staff_names)仅文本存储**：不参与工单/工资的权限过滤
- ✅ 权限过滤已修复：非管理员只能看到自己负责(manager)或创建(created_by)的项目

### 3.12 统计报表模块

**功能逻辑**：
- Dashboard：今日/本月数据、快捷入口、待办提醒
- 基础统计(/api/statistics)：按时间段汇总工单数、金额、收款、支出
- 高级统计(/api/statistics/advanced)：按类型/状态/客户/员工分组统计
- 利润统计(/api/statistics/profit)：收入-成本=毛利/净利（需admin）

**问题**：
- 🔴 **利润统计数据不完整**：只统计了WorkRecord相关数据，**完全没有包含Project模块数据**，利润数字严重失真
- 🔴 **利润计算口径问题**：
  - 从WorkRecord.labor_fee（人工费报价）作为人工成本，而不是实际结算的SalaryRecord.payable_amount
  - 没有从PaymentRecord统计实际收入，而是用WorkRecord.total_fee（合同金额）
- 🟡 **按日期分组有SQL注入风险**：group_by参数直接拼入查询（虽然固定枚举，但应做白名单校验）
- 🟡 **导出Excel功能**：只导出WorkRecord，不包含项目数据
- ✅ Worker权限过滤已应用：只能看到自己参与工单的统计

### 3.13 工单模板模块（WorkTemplate）

**功能逻辑**：
- 保存常用工单内容为模板（适用类型、子类型、工作内容、费用、人员等）
- 创建工单时可"应用模板"快速填充
- 支持公有/私有模板(is_public)

**现状**：
- ✅ CRUD功能完整
- ✅ 权限已修复：创建/编辑/删除需admin
- 🟡 模板字段不完整：没有保存故障相关字段（fault_description/diagnosis等），维修模板实用性降低

### 3.14 操作日志与数据备份

**功能逻辑**：
- 操作日志(OperationLog)：记录增删改操作，保存修改前后快照(snapshot_before/after)，支持回滚(restore)
- 备份功能：手动创建SQLite数据库备份文件，可下载/恢复

**问题**：
- 🟡 **日志回滚功能风险高**：restore接口直接用旧快照覆盖，没有二次确认，可能覆盖后续其他修改
- 🟡 **日志清理(/operation-logs/cleanup)**：清理旧日志但没有备份归档机制
- 🟡 **自动备份缺失**：只有手动备份，没有定时自动备份
- ✅ 操作日志权限：仅admin可见

### 3.15 消息通知模块（Notification）

**现状**：
- ✅ 数据模型完整(Notification)
- ✅ API完整：列表、未读数、标记已读、标记全部已读、清除已读
- ✅ 前端通知下拉菜单和消息中心页面已实现
- ❌ **业务逻辑完全未接入**：_create_notification()辅助函数已定义但**全系统没有任何地方调用它**
  - 新待办指派时不发通知
  - 工单状态变更不发通知
  - 收款/支出/工资操作不发通知
  - 低库存预警不发通知
  - 巡检待办生成不发通知

## 七、代码质量与可维护性问题

### 7.1 后端
