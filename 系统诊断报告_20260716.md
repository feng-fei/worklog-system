## 第一部分：系统大框架

### 1.1 业务定位

这是一个**中小型工程服务公司的业务管理系统**。公司的业务模式是：

> 客户有施工/维修需求 → 接单派工 → 员工上工 → 记录费用 → 收款结算

系统覆盖了从客户报修、工单派工、物料管理、费用记录到收款结算的全流程。

### 1.2 核心业务流程

```
        客户报修/需求
              │
              ▼
    ┌─── 待办事项 ────── 工单模板 ────── 巡检计划 ───┐
    │   (记录需求)       (快速开单)       (定期生成)    │
    │         │              │               │         │
    │         ▼              ▼               ▼         │
    │  ┌──────────────────────────────────────────┐    │
    │  │           工单（施工记录 / 维修工单）      │    │
    │  │   客户信息 → 工作内容 → 派工 → 费用 → 完工  │    │
    │  └──────────────────────────────────────────┘    │
    │         │                                        │
    │         ├────── 扣减物料库存 ──── 物料管理        │
    │         ├────── 记录设备档案 ──── 设备管理        │
    │         ├────── 生成工资记录 ──── 工资管理        │
    │         └────── 产生应收款 ────── 收款管理        │
    │                                                  │
    └──────────────────────────────────────────────────┘
                              │
                              ▼
              ┌──────────────────────────────┐
              │         财务管理              │
              │  收款 ── 支出 ── 工资 ── 利润  │
              └──────────────────────────────┘
                              │
                              ▼
              ┌──────────────────────────────┐
              │         统计与报表            │
              │  仪表盘 ── 统计 ── 导出 ── 日历│
              └──────────────────────────────┘
```

### 1.3 数据模型全景

```
                              ┌────────────┐
                              │  Customer  │ 客户
                              └─────┬──────┘
                                    │ (customer_name 文本匹配，无FK)
        ┌───────────────┬───────────┼───────────┬───────────────┐
        ▼               ▼           ▼           ▼               ▼
  ┌───────────┐  ┌───────────┐ ┌──────────┐ ┌──────────┐ ┌───────────┐
  │ PendingWork│  │WorkRecord │ │ Project  │ │ Payment  │ │CustomerEq │
  │   待办     │  │  工单 ★   │ │  项目    │ │  收款    │ │ 设备档案  │
  └───────────┘  └──┬──┬──┬──┘ └────┬─────┘ └──────────┘ └───────────┘
                    │  │  │         │
        ┌───────────┘  │  │  ┌──────┼──────┐
        ▼              │  │  ▼      ▼      ▼
  ┌───────────┐        │  │ ┌──────┐┌──────┐┌──────┐
  │RepairEquip│        │  │ │ProjWR││ProjEx││ProjSa│
  │ 设备明细  │        │  │ │项目工单││项目费用││项目工资│
  └─────┬─────┘        │  │ └──────┘└──────┘└──────┘
        │              │  │
        ▼              │  │
  ┌───────────┐        │  │
  │ Material  │←───────┘  │  MaterialStockLog 库存流水
  │  物料库存  │          │
  └───────────┘          │
                         ▼
                  ┌───────────┐
                  │SalaryRecord│

                  └───────────┘
```

**关键发现**：整个数据库只有 1 个显式外键约束（`RepairEquipment.work_record_id`），其余 20+ 个关联全靠应用层代码维护。删除工单时不会自动删除关联的工资、收款、设备明细、库存日志。

### 1.4 系统当前状态

| 维度 | 现状 |
|------|------|
| 后端 API | 122 个端点，覆盖 21 个功能模块，功能完整 |
| 前端页面 | 19 个 Tab 页 + 16 个弹窗，功能丰富但入口分散 |
| 前端代码 | index.html(319KB) + custom.js(282KB) + custom.css(50KB)，单体文件过大 |
| 数据模型 | 23 个表，但几乎无数据库级约束，全靠应用层维护 |
| 代码质量 | 11 个 IIFE 模块，v1/v2 两套代码已止血清理 |

## 第二部分：逐一诊断子版块

### 2.1 工单管理

**功能完整性**：✅ 完整

- 施工记录 + 维修工单两种模式
- 50 个字段覆盖客户、工作、人员、费用、维修、反馈、附件
- 支持照片上传（带水印）、临时工管理、动态费用明细表

**表单字段清单**：

| 分组 | 字段数 | 必填字段 | 问题 |
|------|--------|---------|------|
| 客户信息 | 5 | customer_name | 电话、地址非必填但实际业务需要 |
| 工作信息 | 9 | 0 | 日期、工时非必填，不填也能提交 |
| 工作人员 | 2 | staff_names | 临时工面板默认隐藏 |
| 工作内容 | 4 | work_content（仅施工） | 维修模式完全不需要填内容 |
| 费用收款 | 12 | 0 | 金额 0 也能提交 |
| 维修详情 | 14 | 0 | 6 个时间字段全非必填 |
| 客户反馈 | 2 | 0 | 满意度非必填 |
| 备注附件 | 2 | 0 | 照片非必填 |

**逻辑问题**：

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 税额双重叠加 | 🔴 严重 | `create_record()` 算一次，`_sync_equipment_details()` 又算一次 |
| 费用被覆盖 | 🔴 严重 | `material_fee` 被 `equipment_fee_total` 强制覆盖 |
| 新旧费用系统并存 | 🟡 中等 | `fee_items`（JSON动态表）和 6 个 hidden 旧字段共存 |
| 工资全量删除重建 | 🔴 严重 | 每次更新工单，已结算工资丢失状态 |
| 50 字段仅 2 必填 | 🟡 中等 | 数据质量无保障 |
| 维修 6 时间字段无人填 | 🟢 低 | 应精简为 2 个（开始维修、结束维修） |

**优化建议**：
- 必填字段增加到 5-6 个：`customer_name`、`staff_names`、`work_date`、`work_content`（施工）、费用至少一项
- 维修时间精简：6 个 → 2 个
- 彻底移除旧的 6 个 hidden 费用字段，统一用 `fee_items`

### 2.5 财务管理

| 问题 | 严重度 | 说明 |

| 问题 | 严重度 | 说明 |

### 2.6 项目管理

**功能完整性**：⚠️ 功能多但和全局体系割裂

**数据割裂问题**：

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 项目工单与全局工单独立 | 🔴 严重 | `ProjectWorkRecord` vs `WorkRecord`，两套表 |
| 项目费用与全局支出不互通 | 🔴 严重 | `ProjectExpense` vs `Expense` |
| 项目工资与全局工资不互通 | 🔴 严重 | `ProjectSalary` vs `SalaryRecord` |
| 删除项目不级联删除 | 🔴 严重 | 关联的工单/费用/工资残留 |
| 项目数据不入统计 | 🔴 严重 | 仪表盘、统计报表看不到项目数据 |

**这意味着**：项目做完了，全局统计看不到，财务数据碎片化。

| 物料模糊匹配 | 🟡 中等 | 可能扣错库存 |
| 库存不足无感知 | 🟡 中等 | 负数变 0，不告警 |

### 3.3 表单层面

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 工单 50 字段仅 2 必填 | 🟡 中等 | 数据质量无保障 |
| 维修 6 时间字段无人填 | 🟢 低 | 应精简为 2 个 |
| 新旧费用系统并存 | 🟡 中等 | `fee_items` 和 6 个 hidden 字段 |
| 客户缺类型/开票信息 | 🟡 中等 | 影响报价和收款 |
| 收款缺凭证号/收款人 | 🟢 低 | 对账不方便 |
| 支出缺项目关联 | 🟡 中等 | 项目采购无法归集 |

### 3.4 UI/UX 层面

| 问题 | 严重度 | 说明 |
|------|--------|------|

## 第四部分：优化路线图

### 第一阶段：修复数据错误（预计 5 个工时）

**目标**：修复影响财务数据准确性的 bug

| 序号 | 任务 | 文件 | 说明 |
|------|------|------|------|
| 1 | 修复税额双重叠加 | `routes.py` | `_sync_equipment_details` 不再重复计算税额 |
| 2 | 修复费用被覆盖 | `routes.py` | `material_fee` 不再被 `equipment_fee_total` 覆盖 |
| 3 | 修复 paid_amount 两个入口 | `routes.py` + `index.html` | 编辑工单时禁止直接修改 `paid_amount` |
| 4 | 修复工资全量删除 | `routes.py` | 改为增量更新，对比变化 |
| 5 | 修复仪表盘权限 | `routes.py` | `today_total` 和 `unpaid_amount` 加上权限过滤 |

### 第二阶段：打通数据孤岛（预计 3 个工时）

**目标**：项目数据和全局数据互通

| 序号 | 任务 | 说明 |
|------|------|------|

### 核心表（全局工单体系）

```
WorkRecord（工单）★ 核心
├── project_id → Project.id（软外键）
├── 被 RepairEquipment.work_record_id 引用（唯一显式 FK）
├── 被 PaymentRecord.record_id 引用（软外键）
├── 被 SalaryRecord.business_record_id 引用（软外键）
├── 被 PendingWork.related_record_id 引用（多态）
├── 被 RecordEditLog.record_id 引用（软外键）
└── 被 OperationLog.target_id 引用（多态）

Customer（客户）
└── 通过 customer_name 文本字段被 WorkRecord/PendingWork/Project/Payment 引用

Staff（员工）
└── 通过 staff_name 文本字段被 WorkRecord 引用
└── WorkerUser.staff_id 软关联

Material（物料）
├── 被 MaterialStockLog.material_id 引用（软外键）

CustomerEquipment（客户设备）

### 项目独立体系（与全局隔离）

```
Project（项目）

### 关联类型总结

| 类型 | 数量 | 说明 |
|------|------|------|
| 显式 `db.ForeignKey` | 1 个 | `RepairEquipment.work_record_id → work_records.id` |
| 软外键（Integer 字段） | 20+ 个 | 无数据库级约束，全靠应用层维护 |
| 文本匹配关联 | 多处 | `customer_name`、`staff_name` 等文本字段关联 |
| 级联删除（CASCADE） | 0 个 | 没有任何模型设置级联删除 |
